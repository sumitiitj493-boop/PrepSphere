System Design Concepts

13 Core System Design Concepts — The #1 skill that separates senior from junior engineers. Master these, and you can design any system an interviewer throws at you.

Scalability — Scale Up vs Scale Out

Core 4 min
Scalability means your system can handle growing load. Vertical scaling (scale up) = bigger machine. Horizontal scaling (scale out) = more machines. Every modern system at scale goes horizontal.

Vertical Scaling (Scale Up): Add more CPU, RAM, SSD to one server. Simple but has a hard limit — no single machine handles Google-level traffic. Also a single point of failure.

Horizontal Scaling (Scale Out): Add more servers behind a load balancer. Each server is cheap/commodity hardware. If one dies, others handle traffic. This is how Netflix, Uber, Amazon scale.

Why interviews love this: Every system design question starts with "how would you scale this?" The answer always involves horizontal scaling.

Key Interview Points

  • Horizontal scaling is the industry standard for large systems
  • Vertical scaling has a ceiling — can't keep buying bigger machines forever
  • Horizontal scaling introduces complexity: load balancing, data consistency, distributed logic
  • Most systems start vertical, then go horizontal when costs don't justify bigger hardware
  • Auto-scaling (AWS ASG, Kubernetes HPA) adds/removes servers based on traffic automatically

Competitive Edge

Know stateless vs stateful servers. Stateless servers store no user session data — any server handles any request. Stateful servers store sessions in memory, making horizontal scaling harder. Always store sessions externally (Redis, database) so servers stay stateless and trivially scalable.

Follow-up Questions

What happens when one server in a horizontally scaled system crashes?
Load balancer detects via health checks, stops sending traffic, distributes load among remaining servers. Auto-scaler spins up a replacement. Users shouldn't notice.
When would you choose vertical over horizontal?
When workload isn't parallelizable (single massive database needing more RAM), or early-stage when one powerful server is cheaper than managing 10 small ones.

Load Balancer — Traffic Distribution

Important 5 min
A load balancer sits between users and servers. It distributes requests so no single server gets overwhelmed. It's the backbone of every scalable system.

How it works: Client → Load Balancer → picks a backend server → forwards request → returns response. Client never knows which server handled it.

Algorithms:

Round Robin: Next server in order. Simple, doesn't account for server capacity.

Least Connections: Server with fewest active requests. Smart for varying request durations.

IP Hash: Hash client IP → always route to same server. Enables sticky sessions.

Weighted Round Robin: More powerful servers get proportionally more traffic.

Key Interview Points

  • Round Robin is simplest but doesn't adapt to server capacity
  • Least Connections is best for varying request durations
  • IP Hash enables session affinity (sticky sessions) without external session store
  • Health checks: LB pings each server every few seconds, removes unhealthy ones from pool
  • Nginx, HAProxy, AWS ALB are the most common load balancers

Competitive Edge

Know Layer 4 vs Layer 7 load balancing. L4 routes based on IP+port — fast but dumb. L7 routes based on URL path, headers, cookies — e.g., /api/* → API servers, /images/* → CDN. Nginx and HAProxy do both. Cloud load balancers (AWS ALB) are Layer 7.

Follow-up Questions

What if the load balancer itself crashes?
Single LB = single point of failure. Use multiple LBs in active-passive or active-active with a virtual IP. Cloud providers (AWS ALB) handle this automatically — they're inherently highly available.
How does health checking work?
LB periodically sends HTTP/TCP probes (e.g., GET /health every 10s). If server fails 3 consecutive checks → marked unhealthy → removed from pool. When it recovers → added back automatically.

Caching — Making Things Fast

Important 5 min
Caching stores frequently accessed data in fast RAM instead of slow disk/database. Single biggest performance optimization. Redis responds in <1ms vs 50-100ms for database.

Cache Hierarchy (fastest to slowest):

1. Browser Cache: Static files cached in user's browser. Controlled by Cache-Control headers.

2. CDN Cache: Content at edge servers worldwide. User in Mumbai gets content from Mumbai, not US.

3. Application Cache (Redis/Memcached): Database results, computed data, sessions in memory. Sub-millisecond reads.

4. Database Buffer Pool: Internal database cache. Not user-managed.

Cache Strategies:

Cache-Aside (Lazy Loading): Check cache → miss → fetch DB → store in cache → return. Most common.

Write-Through: Write to cache AND DB simultaneously. Always consistent, slower writes.

Write-Behind: Write cache immediately, async write DB later. Fast writes, risk of data loss.

Key Interview Points

  • Every cache entry needs TTL (Time To Live) — without it, stale data lives forever
  • Cache invalidation is one of the hardest problems in computer science
  • Redis: supports data structures, persistence, pub/sub. Default choice for most apps
  • Memcached: simpler, multithreaded, pure key-value. Good for simple caching
  • Cache hit ratio above 90% = your cache is working well

Competitive Edge

Know cache invalidation strategies: (1) TTL-based — expire after time, (2) Event-based — publish event on data change, listeners invalidate, (3) Versioned keys — include version in key (user:v3), change version on update. This is one of the top 3 system design interview topics.

Follow-up Questions

What is cache stampede / thundering herd?
Popular cache key expires → thousands of requests simultaneously hit database. Solutions: (1) Locking — first request locks, others wait, (2) Early expiration — refresh before TTL ends, (3) Never expire — update async on data change.
Redis vs Memcached?
Redis: data structures, persistence, pub/sub, replication. Use when you need features beyond key-value. Memcached: simpler, multithreaded, pure caching. Most new systems use Redis.

Database Scaling — Sharding & Replication

Important 6 min
Single database = bottleneck. Read more efficiently (replication) or split data across machines (sharding). Every large-scale system uses both.

Replication (Read Scaling): Copy data to multiple servers. One primary (writes) → many replicas (reads). 90% of queries are reads — handles most load.

Sharding (Write Scaling): Split data into chunks across servers. Users A-M → Shard 1, N-Z → Shard 2. Each shard handles its own reads AND writes.

Key Interview Points

  • Primary-Replica: All writes to primary, replicated to replicas asynchronously. Reads from any replica
  • Sharding Key: Field used to split data (user_id, region). Wrong choice = hotspots (uneven data)
  • Consistent Hashing: Distribute data evenly, minimize reshuffling when servers change
  • Vertical Partitioning: Split by columns (profile vs settings). Different from sharding
  • Federation: Split by function — user DB, order DB, payment DB. Each scales independently

Competitive Edge

Know when NOT to shard. Premature sharding adds massive complexity (cross-shard queries, transactions, migrations). Start with replication + read replicas + caching. Shard ONLY when write throughput exceeds one primary. Instagram sharded at 1M+ writes/sec. Most apps never need it.

Follow-up Questions

What happens when a shard goes down?
That shard's data is unavailable. Each shard should have replicas for failover. Application-level retry to backup. This is why sharding complexity isn't worth it until absolutely needed.
How do cross-shard queries work?
Avoid them. Design shard key so related data lives on same shard. If needed: query each shard in parallel, merge results (scatter-gather). Slow — minimize.

Consistent Hashing

Concept 4 min
Distributed hashing that minimizes data movement when servers change. Used by every distributed database (Cassandra, DynamoDB) and cache system.

The Problem: Simple hash (hash(key) % N) remaps ~ALL keys when N changes. With 100 servers, adding 1 moves ~99% of data.

Solution: Both servers and keys hash onto a ring (0 to 2³²-1). Each key → first server clockwise. When a server joins/leaves, only adjacent keys move. ~K/N keys remap instead of ~all.

Key Interview Points

  • Places servers and data on a virtual ring (hash ring)
  • Adding server: only keys from its immediate neighbor move
  • Removing server: keys go to next server clockwise
  • Virtual Nodes (VNodes): Each physical server maps to multiple ring positions for even distribution
  • Used by: DynamoDB, Cassandra, Riak, Akamai CDN, Discord

Competitive Edge

Know virtual nodes. Without them, 3 servers can't be evenly utilized. VNodes let powerful servers own more ring segments = more data = more load they can handle. Also ensures better distribution with few physical servers.

Follow-up Questions

Why not hash(key) % number_of_servers?
When server count changes, nearly ALL keys move. Consistent hashing only moves K/N keys. 1M keys, 100 servers: adding one moves ~10K keys instead of ~990K.
How does DynamoDB use this?
DynamoDB partitions data using consistent hashing internally. Each partition handles a hash key range. When you provision more capacity, AWS adds partitions seamlessly.

Message Queues & Async Processing

Important 5 min
User uploads video → can't wait 10 minutes for processing. Instead: put message on queue, return immediately, workers handle it async. Fundamental to every large-scale system.

Producer → Queue → Consumer. Producer creates message, queue stores it, consumer processes it. They never interact directly.

Examples: YouTube: upload → queue → transcode → queue → thumbnails → queue → notify. Uber: ride request → queue → match driver. Every notification system: event → queue → email/push/SMS.

Key Interview Points

  • Decoupling: Producer and consumer don't know about each other — just the queue
  • Buffering: Queue absorbs traffic spikes. Consumers process at their own pace
  • Reliability: Messages persist on disk. Consumer crash → message retried by another
  • Scaling: Add more consumers = faster processing. Queue handles distribution
  • Popular: RabbitMQ (simple, feature-rich), Kafka (high throughput, streaming), SQS (AWS managed)

Competitive Edge

Know Kafka vs RabbitMQ. Kafka is an event streaming platform — messages persist and replay (commit log). Multiple consumers independently read same events. RabbitMQ is traditional queue: message delivered and acknowledged once. Kafka for event sourcing, analytics, audit. RabbitMQ for task queues, notifications.

Follow-up Questions

What if consumer crashes mid-processing?
Message stays 'in-flight' with visibility timeout. If not acknowledged in time, becomes available for another consumer. 'At-least-once delivery' — may process twice, so consumers must be idempotent.
How to ensure exactly-once processing?
True exactly-once is very hard. Most systems use at-least-once + idempotent consumers. Use unique message IDs, check if already processed before handling.

Microservices vs Monolith

Core 5 min
Monolith = entire app in one codebase. Microservices = split into independent services, each with own DB, deployed separately. Start monolith, extract services as you scale.

Monolith: Simple to build/test/deploy initially. As codebase grows, every change requires full redeploy. One bug can crash everything.

Microservices: Each service owns its data, deploys independently. Team A owns payments, Team B owns users. But introduces network calls, distributed consistency, operational complexity.

Key Interview Points

  • Start monolith when: Small team, early stage, validating product-market fit
  • Go microservices when: Multiple teams, different scale needs, independent deployments required
  • API Gateway: Single entry point. Routes to correct service. Handles auth, rate limiting, logging
  • Service Discovery: Services find each other dynamically (Consul, Eureka, K8s DNS)
  • The tradeoff: microservices solve organizational scaling by introducing technical complexity

Competitive Edge

Know the Strangler Fig Pattern. Don't rewrite monolith overnight. Build new features as microservices, gradually redirect traffic. Over 12-18 months, monolith shrinks as services are extracted. Amazon did this over ~5 years. This is what interviewers want to hear.

Follow-up Questions

How do microservices communicate?
Two ways: (1) Synchronous — REST/gRPC. Simple but creates coupling and cascading failures. (2) Asynchronous — message queues. Decoupled, resilient. Best practice: async for inter-service, sync only for real-time user requests.
How handle distributed transactions?
Saga Pattern. Chain local transactions across services. If any step fails, run compensating transactions to undo. Example: create order → charge payment → reserve inventory → if payment fails, cancel order + release inventory.

API Design — REST & Best Practices

Practical 5 min
APIs are how services communicate. REST is the standard. Good API design separates junior from senior developers. Every backend interview asks this.

REST: Resources are nouns (/users, /orders). HTTP methods are verbs (GET, POST, PUT, PATCH, DELETE). Stateless server.

Key Interview Points

  • Plural nouns: /users (not /user), /users/123 for specific user
  • Status codes: 200=OK, 201=Created, 400=Bad Request, 401=Unauthorized, 403=Forbidden, 404=Not Found, 429=Too Many Requests, 500=Server Error
  • Pagination: Never return all records. Use ?page=1&limit=20 or cursor-based
  • Versioning: /api/v1/users. Breaking changes → new version without breaking old
  • Rate Limiting: Protect API. Return 429 with Retry-After header when limit exceeded

Competitive Edge

Know idempotency. Idempotent call = same result no matter how many times called. GET, PUT, DELETE are idempotent. POST is not. For payments, use idempotency key in header — same key = return previous result. Stripe does this.

Follow-up Questions

PUT vs PATCH?
PUT replaces entire resource. PATCH updates only specified fields. PUT {name:'John'} removes all other fields. PATCH {name:'John'} only changes name.
How handle long-running operations?
Return 202 Accepted with status URL. Client polls until complete. POST /videos → 202 with Location: /videos/123/status → poll until done.

CDN & Edge Computing

Concept 4 min
CDN serves static content from servers physically close to users. Reduces latency from 300ms to 30ms. Every high-traffic website uses one.

How CDN works: Configure domain → point to CDN provider (Cloudflare, CloudFront) → CDN caches content at edge locations worldwide → users routed to nearest edge.

Key Interview Points

  • Cache HIT: Edge has content → instant response
  • Cache MISS: Edge fetches from origin → caches → responds. Next user gets hit
  • Purge: Tell CDN to clear stale content. Minutes to propagate globally
  • Edge Computing: Run YOUR code at edge (Cloudflare Workers, Lambda@Edge)
  • Popular: Cloudflare (free tier!), AWS CloudFront, Akamai, Fastly

Competitive Edge

Know cache-busting. Include hash in filename: main.a3b2c1.js. Deploy new code → hash changes → new URL → CDN treats as new. Old cached version still works. Infinite caching + instant updates.

Follow-up Questions

Should APIs go through CDN?
Yes for GET with public data (product listings). Set short TTL (60s). Skip CDN for personalized/sensitive data. Never cache POST/PUT/DELETE.
CDN vs reverse proxy?
Reverse proxy sits in YOUR datacenter (Nginx). CDN sits in MANY datacenters worldwide. CDN is a distributed reverse proxy network.

Design a URL Shortener

Practical 6 min
The most common system design interview question. If you can design this, you understand encoding, databases, caching, analytics, and scaling in one question.

Requirements: Long URL → short code (abc123). Visit short URL → redirect to original. Track clicks. Handle 100M+ URLs.

Generate Short Code: Auto-increment ID → base62 encode (0-9,a-z,A-Z). ID 1='1', ID 61='Z', ID 1000000='4c92'. 7 chars = 62⁷ = 3.5 trillion unique URLs.

Storage: Key-value store (DynamoDB) — short_code is key, long_url is value. Perfect fit.

Key Interview Points

  • 301 vs 302: 301 = permanent (browser caches, no analytics). 302 = temporary (always hits server, track clicks). Most use 302
  • Cache short→long mappings in Redis. 90%+ popular URLs from cache, never hit DB
  • Analytics: async write click data to queue → process → analytics DB
  • 100M URLs × 200 bytes = ~20GB. Fits on one machine. Scale reads with caching
  • Global latency: CDN/edge for redirects

Competitive Edge

Interviewers love base62 encoding. Counter → deterministic short codes. 7 chars = 3.5 trillion URLs. For distributed systems, range-based ID allocation — each server gets an ID range, eliminating centralized counter bottleneck.

Follow-up Questions

Custom aliases (user picks their code)?
Check if exists. If yes, reject. Limit to premium users — wastes namespace. Store in same KV store.
Prevent malware/phishing?
Run through: Google Safe Browsing API, phishing DB lookup, rate limit per IP, user reporting. Block flagged at redirect level.

Rate Limiting & Throttling

Practical 4 min
Controls how many requests a user makes in a time window. Without it, one malicious user sends 1M requests/sec and takes down your system. Every API needs this.

Algorithms:

Fixed Window: Count in fixed time window (100/minute). Allows burst at boundaries.

Sliding Window: Track last N seconds continuously. More accurate.

Token Bucket: Bucket has N tokens. Each request costs 1. Refills steadily. Allows controlled bursts. Most widely used.

Key Interview Points

  • Token Bucket is the industry standard — AWS, Stripe, GitHub all use it
  • Leaky Bucket: constant processing rate. Smooths traffic, no bursts
  • Rate limit at: API Gateway, Application level, Infrastructure (Nginx/Cloudflare)
  • Return 429 Too Many Requests with Retry-After header
  • Implementation: Redis INCR + EXPIRE. O(1) per request

Competitive Edge

Know distributed rate limiting. 10 servers can't count independently — user could send 100 to each = 1000 total. Use shared Redis counter. At larger scale, dedicated rate limiting service or Redis Cluster.

Follow-up Questions

How does Stripe do it?
Token bucket with burst. Different tiers, different limits. Returns RateLimit-Remaining and RateLimit-Reset headers. 429 with clear retry message. Gold standard.
Auth vs unauth rate limiting?
Unauth: by IP (restrictive, 100/hr). Auth: by user ID (generous, 1000/hr). Always both layers — IP as safety net, user as primary.

Real-Time Systems & WebSockets

Practical 4 min
HTTP is request-response. But chat, live scores, stock tickers need server push. WebSockets provide full-duplex persistent connections for real-time communication.

Options:

Polling: Client asks every N seconds. Wasteful, high latency.

Long Polling: Server holds request until data available. Better but HTTP overhead.

SSE: One-way server push over HTTP. Simple, auto-reconnect. Client can't send back.

WebSockets: Full-duplex persistent TCP. Both sides send anytime. Best for chat, gaming, collaboration.

Key Interview Points

  • WebSocket: HTTP Upgrade → WS protocol. No headers after handshake
  • Perfect for: chat, live sports, gaming, collaborative editing, stock tickers
  • SSE is simpler for one-way push (notifications, feeds) — consider it first
  • Scaling: Each connection holds server memory. Use Redis Pub/Sub between servers
  • Socket.io is the most popular WebSocket library with fallbacks

Competitive Edge

Know horizontal WebSocket scaling. User A on Server 1, User B on Server 2. A sends to B → Server 1 publishes to Redis Pub/Sub → Server 2 receives → delivers to B. This is exactly how Socket.io clustering works.

Follow-up Questions

SSE vs WebSocket?
SSE for server→client only (live scores, notifications). Simpler, works over HTTP, auto-reconnects. WebSocket when client→server too (chat, gaming). Don't over-engineer.
How does WhatsApp scale connections?
Ejabberd (Erlang-based) + custom modifications. Erlang designed for massive concurrency. Federation of servers, custom binary protocol. Purpose-built real-time infrastructure.

CAP Theorem in Practice

Core 5 min
In distributed systems: Consistency, Availability, Partition tolerance — pick 2. Since network partitions WILL happen, you really choose between C and A during partitions. This shapes every database decision.

C: Every read returns latest write. All nodes see same data simultaneously.

A: Every request gets a response, even if data might be stale.

P: System works despite network failures between nodes.

Reality: Partitions are inevitable. P is mandatory. Real choice: during partition, prioritize C or A?

Key Interview Points

  • CP: MongoDB (default), HBase, Redis Cluster. During partition, reject potentially stale reads
  • AP: Cassandra, DynamoDB, CouchDB. During partition, serve reads even if stale
  • CA: Only in single-node DBs (PostgreSQL, MySQL). No partition = no choice
  • Eventual Consistency: If no new writes, all replicas converge. Usually within milliseconds
  • Most modern DBs let you configure per-operation: strong for payments, eventual for likes

Competitive Edge

Know PACELC theorem — CAP evolution. Adds: during normal operation (no partition), tradeoff is Latency vs Consistency. DynamoDB lets you choose per read: 'consistent read' costs more but always right. 'Eventually consistent' is cheaper but may be stale by milliseconds.

Follow-up Questions

How does DynamoDB offer both?
Each read has ConsistentRead parameter. True = read from leader (strong, higher cost). False = read from any replica (faster, cheaper, ~1s stale). Most apps use eventual for 95% of reads.
Why can't you have all three?
2 servers, cable breaks (P). User writes to Server 1. Should Server 2 return old data (available but inconsistent) or refuse (consistent but unavailable)? Can't have both during partition.
